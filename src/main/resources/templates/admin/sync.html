CVE Delta Processing

Fetch CVEs via NVD API (lastModified window), upsert vulnerabilities, evaluate software matches, and upsert alerts.

For bulk ingestion from CVE JSON 2.0 feeds, use “CVE Feed Sync (JSON 2.0)”.

Run delta processing

Typical prerequisites
  • Import your inventory (assets / software) via CSV Import.
  • Sync the CPE dictionary and map vendor/product for higher match quality: CPE Dictionary Sync → Canonical Mapping.
Matching works via canonical IDs / normalized vendor+product first, and also supports exact cpe_name matching as a fallback. Version range checks may become UNCONFIRMED when software version is missing/unparsable.
Fetches CVEs whose lastModified is within (now - daysBack) … now. Smaller values = faster, larger values = more catch-up.
Limits how many CVE items are fetched from the API in one run. If the window contains more items than this cap, rerun with a smaller window or run multiple times.

Result

Interpretation: Fetch/Upsert affects the vulnerability store; Pairs/Alerts reflects the matching pass against your current software inventory.

Metric Value Description
fetched 0 Number of CVE items returned by the NVD API in this run.
vulnerabilitiesUpserted 0 Number of vulnerability rows saved/updated.
affectedCpesInserted 0 Number of affected CPE specs inserted (includes version range metadata).
pairsFound 0 Total software × vulnerability candidate pairs evaluated as non-NO_MATCH.
alertsInserted 0 New alerts created (first time a pair becomes visible).
alertsTouched 0 Existing alerts updated (e.g., refreshed lastSeenAt, certainty/matchedBy updated).
Tip: If you only want to run the matching phase (no CVE fetch), use Recalculate Alerts.
